# Agent Vault

## What This Repository Is

This is a **git-tracked Obsidian vault** serving as a knowledge base. It contains
reference documentation, implementation schemas, code reviews, and triage
entries — not source code. There are no build systems, tests, or linters.

## Vault Structure

### `tasks/<owner>/<repo>/<task>/` — Implementation tasks

Each task directory contains up to three files:

- `schema.md` — Actionable implementation spec that an AI agent can execute step-by-step.
- `review.md` — Structured code review for work done against the schema.
- `triage.md` — Escalation notes, design questions, and run summaries.

### `tasks/_fleet/<task>.md` — Fleet (cross-repo) schemas

Umbrella documents coordinating work across multiple repositories. These are
flat files (not directories) since they don't have per-task reviews or triage.

### `notes/<owner>/<repo>/` — Repository reference documentation

Reference summaries of repository internals (build systems, architecture,
scripts). A future agent session should be able to understand a subsystem
by reading one file.

### `designs/` — Design documents

High-level thinking: architecture explorations, trade-off analyses, roadmaps,
and cross-cutting concerns.

### `drafts/` — Work-in-progress documents

Staging area for notes and specs under active development.

### `_misc/` — Infrastructure (hidden by `_` prefix in Obsidian)

Umbrella directory for all vault infrastructure that is not user-facing content.
Collapsed by default in the Obsidian file explorer. Contains:

- `_misc/archive/` — Completed work
- `_misc/templates/` — Format templates
- `_misc/images/` — Notification icons and image assets

#### `_misc/archive/` — Completed work

Contains archived task directories (schema + review + triage) that have been
completed and moved out of `tasks/`.

When asked to archive tasks, move them from `tasks/` into `_misc/archive/`.

### `audits/<owner>/<repo>/` — Audit reports

Point-in-time quality snapshots generated by `@auditor`. Each file is
named `<date>-<label>.md` (e.g. `2026-03-29-full-audit.md`). Multiple audits
per repo accumulate here as a historical series.

### `projects/<owner>/` — Project status documents

Human-readable per-repo project snapshots maintained by `@project-manager`.
Each file (`projects/<owner>/<repo>.md`) mirrors the current state of open/closed
issues, milestone progress, and project board columns for a vault-managed repo.
Refreshed whenever PM performs state-changing operations or is explicitly asked
to sync. Use `backend: github` for GitHub-backed projects and `backend: local`
for vault-only tracking.

### Other

- `.obsidian/` - Obsidian configuration (do not modify).
- `AGENTS.md` - This file.
- `*/README.md` - Overviews of the various folders and their purposes. Read these for more context.

## Workflow examples

### Building features and bug fixes

1. The user starts in Plan mode. Through exploration and Q&A, a schema is created in the vault. During this process an issue is posted to GitHub or another tracking service.
2. A new context is created in Build mode. The builder implements the defined schema, either using the @implementor or the auto-implement skill. During this process, @reviewer subagents are spawned to check code quality.
3. A final check is done with the help of Audit mode before a PR is pushed and reviewed.

### Auditing code

1. The user starts in Audit mode. A prompt is given, typically to assess the code for security vulnerabilities, code smell, or maintainability issues.
2. The audit is performed, usually with the help of @auditor, though sometimes manually. An audit is produced in the vault.

### Designing features

1. The @designer is called, often in Plan mode, to help plan out a feature. Designs are less implementation-heavy than tasks, and are useful for large-scoped features which cannot be tackled in a single implementation cycle.

### Vault management

Any mode is capable of managing the vault through the vault skills. Typical actions are to view the current vault status, clean up remaining tasks, or to call @project-manager to create an attack plan.

## Agent Roles

| Agent | Role |
| ----- | ----- |
| `@planner` | Schema authoring, GitHub issue creation |
| `@project-manager` | Issue lifecycle, project board ops, milestone management, vault project status |
| `@implementor` | Single commit-group schema executor. |
| `@reviewer` | Structured code review |
| `@designer` | Repo notes, design documents, drafts |
| `@auditor` | Structured audit agent, analagous to @planner. Tool detection, static analysis, vault report |

## Conventions

- Documents are Markdown, authored for Obsidian (may use wiki-links, callouts).
- Do not hard-wrap prose — Obsidian soft-wraps paragraphs.
- Schemas and reviews use YAML frontmatter for metadata (`repo`, `status`, `date`, etc.).
- Path convention: `tasks/<owner>/<repo>/` mirrors the local checkout path
  convention `~/repos/<owner>/<repo>`.
- Agents interact with the vault via the vault tools.
- **Task and issue titles must not contain regex special characters** (`.`, `(`, `)`, `*`, `[`, `]`, `+`, `?`, `{`, `}`, `^`, `$`, `|`, `\\`).^[1]

^[1]: The `vault-lint` script uses title text as a regex pattern when scanning review files — special characters will silently corrupt issue-block extraction.
